Introduction

Project Description Overview

PTA Pilot is a hackathon MVP for an AI PTA communications agent. It models a weekly communications workflow for a PTA VP of Communications across Gmail, Membership Toolkit, board review, teacher release, and parent scheduling. The application is built to remain demoable in mock mode while still supporting a live delegated-access story through Auth0.

Purpose

PTA communication work is repetitive and spread across multiple channels: reminder emails, parent replies, teacher notes, board review, newsletter edits, publishing, and scheduled parent sends. PTA Pilot turns that into a structured workflow with a visible planner, editable newsletter drafts, integration status, and explicit approval gates for risky steps.

What the current repo demonstrates

  • A dashboard shell with Setup, Inbox, Newsletter Editor, Actions Review, and Audit Log views
  • Mock WhatsApp and iMessage ingestion plus Gmail reminder-thread support
  • Structured extraction into newsletter-ready content items
  • Board and teacher draft editing, with parent drafts derived from the teacher-approved path
  • Approval-gated Gmail send, Membership Toolkit publish, and Membership Toolkit schedule flows
  • Audit history for suggestions, approvals, execution steps, and ingestion events

Outputs visible in the product today

  • Board review draft newsletter
  • Teacher edition newsletter draft
  • Derived parent newsletter draft
  • Approval cards for Monday, Wednesday, Thursday, and Sunday actions
  • Flyer recommendations for visually strong or urgent event items
Key features

Current Product Capabilities

1. Setup and integrations The Setup view stores Auth0 account email, PTA contacts, member recipients, school breaks, and integration modes for Auth0, Gmail, Membership Toolkit, mock messages, and flyer generation.
2. Multi-source inbox PTA Pilot ingests Gmail reminder-thread replies, mock WhatsApp messages, mock iMessage messages, previous-newsletter links, and OCR-extracted calendar screenshots.
3. Structured extraction Inbox items are converted into typed extracted content items with a title, summary, source reference, priority, recommended placement, and flyer recommendation flag.
4. Newsletter drafting The newsletter engine can duplicate the last newsletter, build sections from extracted content, keep urgent items first, and keep time-sensitive items above evergreen content.
5. Flyer recommendation PTA Pilot identifies candidate items for flyer treatment and creates a school-friendly brief. In mock mode it also assigns a placeholder image URL.
6. Approval-gated execution Every send, publish, or schedule action is modeled as an approval item with execution steps and visible status transitions.
7. Live readiness visibility The UI exposes Gmail connect status, required versus granted Gmail scopes, Management API readiness, Token Vault readiness, and Gmail API probe results.
Architecture

System Architecture

PTA Pilot is implemented as a monorepo with clear frontend, backend, and shared-domain boundaries. The current architecture is demo-first, with local JSON persistence as the happy path and Firestore marked as the live-mode direction.

Frontend Next.js App Router app in apps/web
Backend Express + TypeScript API in apps/api, intended for Google Cloud Run
Shared package Typed models, Zod schemas, and seeded demo data in packages/shared
Current persistence Local JSON runtime state in apps/api/data/runtime-state.json
Live data direction Firestore for workflow state and logs

Frontend responsibilities

  • Dashboard UI and workflow navigation
  • Auth0 session handling when env vars are present
  • Proxying /api/* calls to the backend through PTA_API_BASE_URL
  • Guest-mode fallback when Auth0 is not configured

Backend responsibilities

  • Server-side token handling and Gmail integration
  • Membership Toolkit adapter selection
  • Inbox artifact storage and OCR
  • Newsletter generation, approval execution, and audit logging
  • Gemini integration and deterministic fallback behavior

Workflow and control model

Data Flow, Planner Logic, and Approval Boundaries

Weekly workflow stages

  • Monday reminder
  • Collect updates
  • Wednesday board draft
  • Thursday teacher release
  • Sunday parent schedule

Data flow in the current app

  1. Setup defines contacts, member recipients, school breaks, and integration modes.
  2. Inbox collects Gmail replies, mock messages, previous-newsletter links, and OCR-extracted artifacts.
  3. Extraction produces typed content items with priority and placement hints.
  4. Newsletter rules build board and teacher drafts from extracted items.
  5. Parent newsletters are derived downstream and remain read-only in the editor.
  6. Flyer logic recommends visual treatment for suitable items.
  7. Approval actions are created for send, publish, and schedule steps.
  8. The backend writes audit entries for suggestions, approvals, ingestion, and execution.

Approval boundaries in the code

  • Monday reminder email send
  • Wednesday board review email send
  • Thursday teacher version publish
  • Sunday parent version scheduling
Why this matters for the hackathon. The official judging criteria emphasize explicit permission boundaries and user control. PTA Pilot surfaces risky actions as approval cards, shows scope status in the UI, and keeps delegated access server-side.
Installation

Local Development Setup

Base setup

  1. Install dependencies with pnpm install.
  2. Copy values from .env.example into apps/web/.env.local and apps/api/.env.
  3. Start both apps with pnpm dev.
  4. Open http://localhost:3000.

Current local ports

  • Web: http://localhost:3000
  • API: http://localhost:8081

Verified commands recorded in the repo README

  • pnpm typecheck
  • pnpm --filter web lint
  • pnpm --filter web build
  • pnpm --filter api build
  • pnpm dev

The README also records a successful local smoke test returning 200 OK from http://localhost:3000 and a healthy JSON response from http://localhost:8081/health.

Demo readiness characteristics

  • Mock mode works end to end out of the box.
  • Guest/demo mode remains usable even if Auth0 is not configured.
  • Vertex AI is optional for the demo story because deterministic fallbacks exist.

Live mode prerequisites

Auth0, Gmail, and Membership Toolkit Setup

Auth0 and Gmail path

  1. Create or reuse an Auth0 Regular Web Application.
  2. Set Allowed Callback URLs to http://localhost:3000/auth/callback.
  3. Set Allowed Logout URLs to http://localhost:3000.
  4. Configure tenant domain, client ID, client secret, app base URL, and secret in the web and API env files.
  5. Enable a Google or Google Workspace connection with the Gmail scopes used by the repo.
  6. Configure Auth0 Token Vault for that connection.
  7. Create a Machine to Machine app for the Auth0 Management API.
  8. Provide Management API credentials to the backend env.
  9. Use the Setup screen to connect or reconnect Gmail with the requested connection_scope values.
  10. Enable the Gmail API on the Google Cloud project behind the Auth0 Google connection.

Membership Toolkit modes

  • mock: local JSON-backed behavior for duplicate, update, publish, and schedule
  • manual: browser/manual path behind the adapter boundary
  • live: Playwright-prepared operator session with needs_operator steps
Important implementation detail. The backend contains a server-side Token Vault exchange path and an Auth0 Management API fallback path for delegated Gmail access. For the hackathon, the safest final demo path is validating the Token Vault exchange against the target tenant before submission.

Configuration

Configuration Overview

PTA Pilot is configured through split web and API env files. The repo ships a single .env.example that documents both sections.

Configuration groups

  • Web app routing and Auth0 session configuration
  • API port, CORS, and delegated-access configuration
  • Gmail scope and connection configuration
  • Membership Toolkit live-mode credentials
  • Gemini / Vertex and Firestore settings
  • Execution-mode flags such as MEMBERSHIP_TOOLKIT_MODE and FLYER_MODE
Verified nuance. The code in apps/api/src/config/env.ts expects tokenset-inspection capability in the Auth0 Management API scope set, and the README explicitly calls out read:federated_connections_tokensets during Auth0 setup.

Current operational defaults in code

  • API port defaults to 8081
  • API base URL defaults to http://localhost:8081
  • Google Cloud location defaults to us-central1
  • Vertex model defaults to gemini-3.1-pro-preview
  • Membership Toolkit and flyer modes default to mock
